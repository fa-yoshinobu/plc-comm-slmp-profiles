# plc-comm-slmp-profiles

This repository is the canonical source for PLC model profile data shared by the SLMP communication libraries (dotnet / python / rust / node-red / cpp-minimal). The Markdown definition files are the editable source; the JSON files are generated distribution artifacts imported by implementation repositories by fixed tags.

## Port Scope

These profiles target CPU built-in Ethernet ports. Extension Ethernet modules may support additional commands and therefore can have a larger usable command set than the CPU built-in Ethernet port. Because the built-in Ethernet profile is the more constrained baseline, using a built-in Ethernet profile when accessing an extension Ethernet module is conservative and should not enable commands beyond the stricter built-in Ethernet baseline; it may only leave module-specific capabilities unused.

## Layout

| Path | Contents |
|------|----------|
| `evidence/profile-definitions/*.md` | Editable capability profile definitions. |
| `capability/slmp_builtin_ethernet_profiles.json` | Generated per-profile capability definitions for nine profiles. These policies are scoped to CPU built-in Ethernet ports only and include frame/subcommand groups, feature states, point limits, write policy, and route selection. |
| `device-ranges/slmp_device_range_rules.md` | Editable device-range rule definition. |
| `device-ranges/slmp_device_range_rules.json` | Generated device-range rules for nine profiles. Defines the SD register block location, family resolution rules (`fixed`, `word-register`, `dword-register`, clipped variants, `unsupported`, `undefined`), and runtime probe behavior. |
| `tables/slmp_profile_comparison.md` | Generated maintenance view with full capability and device-range matrices. Do not edit this file manually. |
| `tools/generate_capability_profiles.py` | Generator for `capability/slmp_builtin_ethernet_profiles.json` from `evidence/profile-definitions/*.md`. |
| `tools/generate_device_range_rules.py` | Generator for `device-ranges/slmp_device_range_rules.json` from `device-ranges/slmp_device_range_rules.md`. |
| `tools/generate_profile_tables.py` | Generator for `tables/slmp_profile_comparison.md`. Run it after regenerating either JSON file. |
| `instructions/slmp_profile_restriction_instructions_20260703.md` | Implementation instructions for rolling out the capability profiles to the libraries, including guard behavior, `strict_profile`, and conformance tests. |
| `evidence/` | Live-device verification records from 2026-07-03 for R120P / iQ-L / iQ-F / LCPU / QnUDV. |

## Role Of The Two Generated JSON Files

- **capability** answers "which features may be sent to the CPU built-in Ethernet port for this profile." It is policy data that cannot be reliably discovered at runtime. It does not describe device existence or ranges.
- **device-ranges** answers "how to discover whether device families exist and what their ranges are." It includes runtime probe algorithms for PLCs where some Q-series device ranges are not written into SD registers.

## Profile IDs

Canonical IDs match each implementation's `SlmpPlcProfile` equivalent:
`melsec:iq-r` / `melsec:iq-l` / `melsec:mx-r` / `melsec:mx-f` / `melsec:iq-f` / `melsec:qcpu` / `melsec:lcpu` / `melsec:qnu` / `melsec:qnudv`

The capability table includes all nine profile IDs. `qcpu` and `qnu` are policy profiles that follow the QnUDV built-in Ethernet capability decisions until live evidence supersedes that policy.

## Editing Rules

1. **Do not edit generated JSON by hand.** Update the Markdown definition file, then regenerate JSON and tables.
2. **Do not add unsupported entries.** Every capability change must carry a `source` value (`live` / `manual` / `spec` / `policy` / `inferred`). Add live verification records under `evidence/` when the source is live testing.
3. If live behavior differs from the current generated data, record the result in evidence first and then update the relevant Markdown definition.
4. Publish changes with a commit and a tag. Implementation repositories must state the imported tag and verify conformance with fixture tests.
5. If the schema changes, increment `schema_version` and keep the old tag available until every implementation has migrated.
6. After changing capability definitions, run `python tools/generate_capability_profiles.py`.
7. After changing device range definitions, run `python tools/generate_device_range_rules.py`.
8. After regenerating either JSON file, run `python tools/generate_profile_tables.py` and commit the regenerated comparison table.

## Background

The initial data set was created from live verification of five PLC families on 2026-07-03. Tests used CPU built-in Ethernet ports over TCP 1025.
