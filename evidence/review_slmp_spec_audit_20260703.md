### 修正状況（2026-07-03）

- F-1: dotnet / python / rust の標準拡張デバイス指定を、マニュアル順の Q/L=11バイト・iQ-R=13バイトへ修正済み。D100 および D100Z4 相当の期待フレームをテストで固定。
- F-2: rust の要求データ長を `checked_add` + `u16::try_from` で検証し、65,535バイトを超える要求データ長は送信前にエラー返却するよう修正済み。巨大ラベルpayloadでも長さフィールドが巻かれず、フレーム未送信で止まることをテストで固定。
- F-3: dotnet / python / cpp の拡張ランダム/拡張モニタ登録で、008x 時は 96点 / weighted 960 / 94点を使うよう修正済み。通常ランダムの Legacy 上限 192/1920/188 は維持。rust は現行公開APIにランダム ext 経路がないため F-3 のコード変更なし。
- F-4: dotnet / python / rust / cpp / node-red の direct bit 一括読出/書込で、iQ-Fプロファイルのみ上限3584点、その他プロファイルは7168点を維持するよう修正済み。ワード単位上限960点は変更なし。
- F-5: dotnet / python / rust の4E応答処理で、要求serialと一致しないD4応答を破棄し、次の応答を待つよう修正済み。pythonは同期/非同期の両クライアントで対応。cpp/node-redは修正前から照合済み。
- F-6: python 同期クライアントのUDPソケットを接続済みUDPへ変更し、`sendto/recvfrom` ではなく `send/recv` で送受信するよう修正済み。別ポートの偽送信元から先に届いたUDP応答を採用せず、正しい送信元の応答だけを採用することをテストで固定。
- F-8: dotnet / python / rust / cpp / node-red で、終了コード≠0の応答データ部9バイトを `network/station/module_io/multidrop/command/subcommand` として構造化するよう修正済み。実機で得た `00 FF FF 03 00 01 04 01 00` を使う回帰テストを追加。
- node-red版: 現行公開APIに `readRandomExt` / `writeRandomExt` / 拡張デバイス指定ビルダーがなく、通常ランダムは 0000/0001 または 0002/0003 のみを使うため、F-1/F-3の直接修正対象外。raw `request()` で008xを手組み送信する場合は呼び出し側責任。
- 検証: dotnet full test、python full pytest、rust cargo test、cpp local CI gate、node-red `npm test` はすべて成功。修正後pythonでR120P実機スモーク（4E通常D100、3E通常D100、3E/0080拡張D100、0403/0080 97点異常応答のF-8構造化、read_random_ext 97点送信前拒否）とFX5U実機スモーク（M0 3584点成功、M0 3585点送信前拒否、raw 3600点C051のF-8構造化）も成功。

### R120P実機確認メモ（2026-07-03）

R120P/RCPU の標準実機確認経路は **4Eフレーム + iQ-Rプロファイル + サブコマンド0002/0003/0082/0083** とする。3E/Q-L互換フレームは比較用であり、R120Pの機能可否を不可判定する根拠にはしない。

詳細な実測結果は `r120p_slmp_live_verify_20260703.md` に分離した。要点は以下。

- `0401/1401`, `0403/1402`, `0801/0802`, `0406/1406` はR120P 4E/iQ-Rで広く使用可能。
- direct最大点数は word 960点成功/961点`C051`、bit 7168点成功/7169点`C052`。
- random/monitor境界は random read 96点成功/97点`C054`、random bit write 94点成功/95点`C053`、monitor登録96点成功/97点`C054`。
- `S`書込み、`LTS/LTC/LSTS/LSTC` のbit direct/block経路は `4030`。ただしユーザー向けには専用ロング系経路に集約し、`LTS/LTC/LSTS/LSTC` の状態読出し、`LTN10:D`/`LSTN10:D` 現在値書込み、`LTC10:BIT`/`LSTC10:BIT` コイル書込みは read/write/restore 成功。
- `LCN/LCS/LCC` も専用ロング系経路に集約する。R120Pでは `LCN10:D`/`LCN10:L` の read/write/restore、`LCS10:BIT`/`LCC10:BIT` 状態読出し、`LCC10:BIT` コイル書込みが成功。`LCS` は接点なので今回の正規経路では書込み対象にしない。blockで得たロングカウンタ raw 成功は参考観測であり、マニュアル上の positive path にはしない。
- `Z` は `Z0..Z19` まで。`LZ` は `LZ0/LZ1` だけのロング固定デバイスで、32-bit経路（`:D`/`:L`、random/named dword）として扱う。16-bit direct/block は意味がないため、raw観測があっても必ず positive path にはしない。R120Pでは `LZ0/LZ1` の random dword、named `:D`、named `:L` read/write/restore が成功。
- `standalone G/HG` は仕様上不可。今回のR120P positive pathは `U3E0\G...` / `U3E0\HG...` / `U2\G100`。`U2\G100` は direct、random、monitorで成功。`Un\Gn` はPLCのユニット構成で可否が変わるため、対象PLCごとに毎回実機確認する。
- HGのCPU-buffer経路はiQ-R専用。iQ-F / iQ-L / LCPU / QCPU / QnU / QnUDV / MX系には定義しない。G系の `Un\Gn` ユニットバッファアクセスとは分けて扱う。

### QnUDV内蔵CPU実機確認メモ（2026-07-03）

QnUDV内蔵CPU の標準実機確認経路は **3Eフレーム + Q/L互換プロファイル + サブコマンド0000/0001** とする。R120Pの4E/iQ-R前提を持ち込まない。

詳細な実測結果は `qnudv_slmp_live_verify_20260703.md` に分離した。要点は以下。

- `D9000` direct word、`Y1FFF` direct bit は read/write/restore 成功。
- `D9000` random word、`Y1FFF` random bit は read/write/restore 成功。
- monitor `0801/0802` は `D9000` / `R10` / `ZR10` で成功。monitor登録は192 word成功、193 word rawで `C054`。
- named read `D9000:U` / `Y1FFF:BIT` 成功。
- `0101/0000` Read Type Name は `C059`。このpathでは型名読出しをpositive pathにしない。
- `0401/0080` extended read は `U0\G10` / `U2\G1000` とも `C070`。今回の内蔵CPU設定では `Un\G` 拡張読出しをpositive pathにしない。
- block `0406/1406` は high-level APIで送信前ガードし、raw送信でも `C059`。QnUDV内蔵Ethernet path のtarget/path非対応として扱う。
- direct最大点数は word 960点成功/961点`C051`、bit 7168点成功/7169点`C052`。
- random境界は read 192点成功/193点`C054`、word write weighted 1920成功/1932点相当`C054`、bit write 188点成功/189点`C053`。write境界は現在値をそのまま書き戻すsame-values方式で確認し、意図的な値変更は行っていない。
- `Z` は `Z0..Z19`、`R` は `R0..R32767`、`ZR` は `ZR0..ZR393215` が今回のQnUDV内蔵CPUで使用可。各上限+1はraw direct readで `4031`。`Z10` / `R10` / `ZR10` は direct、random、named `:U` で write/read/restore成功。
- ロングタイマ、ロング積算タイマ、ロングカウンタは今回のQnUDV内蔵CPUではなし。`LTN/LSTN/LCN/LCC` 代表readはいずれも `C05B` のため、R120Pの専用ロング系 positive path をQnUDVへ持ち込まない。write確認は事前readで失敗したため送信未到達。

### MELSEC iQ-F / FX5UC実機確認メモ（2026-07-03）

MELSEC iQ-F / FX5UC の標準実機確認経路は **3Eフレーム + Q/L互換プロファイル + サブコマンド0000/0001** とする。ソフト側プロファイルは `melsec:iq-f` を使う。`X` / `Y` 文字表記は8進であり、他プロファイルの16進表記を持ち込まない。

詳細な実測結果は `iqf_slmp_live_verify_20260703.md` に分離した。要点は以下。

- 型名応答 `0101/0000` は `FX5UC-32MT/D`、model code `0x4A91`。
- `D10` direct/random/named word、`M10` direct/random bit は read/write/restore 成功。`X10` はread成功、`Y10` はwrite/read/restore成功。
- block `0406/1406` は `D10` / `M10` で read/write same-values成功。iQ-Fではblockをpositive pathにする。
- monitor `0801/0802` は `D10` 単独でも `C059`。iQ-Fではmonitorをpositive pathにしない。
- `0401/0080` extended read は初回FX5UC構成では `U0\G0` / `U1\G0` / `U2\G1000` がいずれも `C060`。特殊ユニットありFX5へ交換後、`U1\G0` read は成功。`Un\Gn` はiQ-Fでも構成依存として対象PLCごとに確認する。
- 特殊ユニットあり別品番FX5U-32MR/DSで全項目を再確認した結果、通常機能・境界・範囲・block/monitor可否・未対応デバイスは初回FX5UCと同等。差分は型名と `U1\G0/G1/G10` の成功のみ。`U1\G0/G1/G10` は write/read/restore も成功。詳細は `iqf_slmp_live_verify_20260703.md` に集約。
- HGのCPU-buffer経路はiQ-R専用であり、iQ-Fには定義しない。iQ-Fで採用するのは実在確認できた `U1\G...` のG系ユニットバッファアクセスのみ。
- direct最大点数は word 960点成功/961点`C052`、bit 3584点成功/3585点`C051`。iQ-Fのdirect bitだけ3584点上限を使う。
- random境界は read 192点成功/193点`C054`、word write 160点成功/161点`C054`、bit write 188点成功/189点`C053`。write境界は現在値をそのまま書き戻すsame-values方式で確認し、意図的な値変更は行っていない。
- `X` / `Y` は8進で、`X1777` / `Y1777` は成功、`X2000` / `Y2000` は `C056`。
- `Z` は `Z0..Z19`、`R` は `R0..R32767`、ロングカウンタは `LC0..LC63`、`LZ` は `LZ0/LZ1` が今回のFX5UCで使用可。各上限+1は `C056`。
- `LCN10:D` / `LCN63:D`、`LCC10:BIT`、`LZ0:D` / `LZ1:D` は write/read/restore成功。`LCS` は状態読出し対象。`LZ` は16-bit directではなく32-bit経路（named/random dword）をpositive pathにする。
- `ZR`, `RD`, `V`, `LT`, `LST` は今回のFX5UCでは `C05C` のためpositive pathにしない。

## 発見事項の詳細

### F-1【高】拡張デバイス指定の「標準」ビルダーのバイト順がマニュアルと不一致

**マニュアル根拠**: SH-080003 付1（p.428〜432）および **SH-080931 付1（p.204〜220）— 両マニュアルで同一**。バイナリコード時の拡張指定フォーマットは:

```
[デバイス修飾・間接指定(2)] [デバイス番号(3/4)] [デバイスコード(1/2)] [拡張指定修飾(2)] [拡張指定(2)] [ダイレクトメモリ指定(1)]
→ Q/L系=11バイト、iQ-R系=13バイト
```

実例（p.431、サブコマンド0080でJ1\W100にアクセス）:
`80 00 | 00 00 | 00 01 00 | B4 | 00 00 | 01 00 | F9`

**実装状況**:
- G（DM=F8）、iQ-R専用HG CPU-buffer（DM=FA）、リンクダイレクト（DM=F9）向けの「capture-aligned」ビルダーは上記と**一致**（GOT pcap検証済み・実機ライブ検証済み）。HG CPU-bufferはiQ-R専用であり、他プロファイルには定義しない。
- しかしそれ以外の拡張アクセス（通常デバイス+008x、インデックス修飾、間接指定）に使われる**標準ビルダーは順序も長さも異なる**:

```
実装の標準ビルダー: [拡張指定(2)] [拡張指定修飾(1)] [修飾index(1)] [修飾flags(1)] [デバイス番号+コード(4/6)] [DM(1)]
→ Q/L系=10バイト、iQ-R系=12バイト（マニュアルより1バイト短い・順序逆転）
```

- 該当箇所:
  - dotnet: `SlmpPayloads.cs:311-321`（`EncodeExtendedDeviceSpec`標準分岐）
  - python: `slmp/core.py:942-947`（`encode_resolved_extended_device_spec`標準分岐）
  - rust: `src/client.rs:2142-2149`（`encode_extended_device_spec`標準分岐）
  - cpp: **該当なし**（J\とU\G/HGのみサポートで、両方ともマニュアル準拠レイアウト）
  - node-red: **該当なし**（008x系の拡張APIを公開していない）

**影響**: `read_devices_ext("D100")` のような通常デバイス+008xのアクセス、および`make_extension_spec`によるインデックス修飾/間接指定アクセスは、マニュアル上のフォーマットと異なるフレームを送信する。長さが異なるためPLC側でC051/C05C系エラーになるか、最悪フィールドがずれて解釈される。**2026-07-03実機確認で、現行10バイト標準レイアウトはR120PがC05Bで拒否し、マニュアル11バイトレイアウトは成功したため不一致確定**。

**SH-080931での確定（2026-07-03追加照合）**: SH-080931 付1「デバイス番号をインデックスレジスタ…で間接指定するアクセス」（印刷p.220）のバイナリ図は、リンクダイレクト/ユニットアクセス/CPUバッファ**以外の通常デバイス**についても

```
[デバイス修飾] [先頭デバイス番号またはデバイス番号] [デバイスコード] [拡張指定修飾] [拡張指定] [ダイレクトメモリ指定]
```

の順を明示しており（0080/0081・0082/0083とも同じ）、SH-080003と完全に一致。**両マニュアルとも標準ビルダーの「拡張指定先頭」順とは異なる**。

**背景推定**: internal_docsではcapture-alignedを「reordered（並び替え済み）」の特殊形として扱っているが、両マニュアルの図に照らすと**capture-alignedの方が正規レイアウト**であり、標準ビルダーの方が誤り（おそらくフィールド定義表の記載順をワイヤ順と誤読したもの）。GOT pcapがマニュアル通りだったのはその傍証。

**推奨対応**:
1. ~~SH-080931の該当図でクロスチェック~~ → **完了（不一致確定）**

### F-2【高・発生条件は狭い】Rust: 要求データ長のu16サイレントラップ

`src/client.rs:1833`（4E）/`1842`（3E）:

```rust
frame[11..13].copy_from_slice(&(6u16 + payload.len() as u16).to_le_bytes());
```

ペイロードが65,529バイトを超えると `as u16` が黙って切り捨て、**長さフィールドだけ壊れたフレームを送信**する。デバイス系コマンドは点数上限で守られているが、**ラベル系（`write_random_labels`等）は個々のdata長(≤65535)しか検証しておらず合計長のチェックがない**ため到達可能。

他実装は保護あり: dotnet=`checked()`で例外、python=`_check_u16`、node-red=`ensureRange`、cpp=固定txバッファの容量チェック。

**修正（2026-07-03）**: `request_data_length()` を追加し、`payload.len() + 6` を `checked_add` + `u16::try_from` で検証してから3E/4Eヘッダへ書き込むよう修正済み。上限超過時は `SlmpError` を返し、`last_request_frame` と送信カウントは更新されない。

**回帰テスト**: `write_random_labels` に 65,520バイトのラベルデータを渡し、個別data長はu16範囲内だが要求データ長合計が65,536バイトになるケースを追加。修正後は送信前に `request data length must be <= 65535 bytes` で失敗する。

### F-3【中】拡張サブコマンド(008x)時の点数上限が非拡張の値のまま

**マニュアル根拠**: SH-080003 §8.3/8.4 および FX5 SH-082624 §38.1。上限は**シリーズではなくサブコマンドで決まる**:

| コマンド | 0000/0001系 | 0080/0081/0082/0083系 |
|---|---|---|
| 0403 ランダム読出 | 192点 | **96点** |
| 1402 ワードランダム書込 | weighted≤1920 | **weighted≤960** |
| 1402 ビットランダム書込 | 188点 | **94点** |
| 0406/1406 ブロック数 | 120 | **60** |

**実装状況**: 全実装が「Legacy(Q/L)=大きい方、iQR=小さい方」というモード基準で検証している。
- iQ-Rモード: 非拡張=0002/0003(上限96/94/960)、拡張=0082/0083(同じ上限) → **偶然一致で正しい**
- **Legacyモードの拡張API**（`ReadRandomExtAsync`等が0080/0081を送る）: 192/188/1920まで通してしまう → マニュアル上限96/94/960超過の要求をPLCに送りうる

該当: dotnet `SlmpClient.cs:1546-1573`、python `core.py:1066-1092`、rust `client_rules.rs:45-106`、cpp `slmp_minimal.cpp:171-179`。node-redは拡張API非公開のため実害なし。ブロック系は拡張API未公開のため現状実害なし。

**推奨対応**: 検証関数に「拡張かどうか」を渡し、008x時は96/94/960を適用。

**実機確認（2026-07-03）**: R120P / 192.168.250.100:1025 TCPで、0403/0080ランダム読出しをマニュアル順11バイト拡張指定で送信。96ワード（D0〜D95）は終了コード0000、97ワード（D0〜D96）は終了コードC054。マニュアル上限96点と一致し、現行Legacy拡張APIが97点以上を送信できる場合はPLC側拒否となる。

**修正（2026-07-03）**: dotnet / python / cpp は、通常ランダムのLegacy上限を維持したまま、拡張ランダム/拡張モニタ登録のみ008x上限（96/94/960）で検証するよう修正済み。rust は現行公開APIにランダム ext 経路がないため変更なし。

**node-red版の扱い（2026-07-03確認）**: `lib/slmp/client.js` の `readRandom` / `writeRandomWords` / `writeRandomBits` は拡張なしの `resolveDeviceSubcommand` を使い、0080/0081/0082/0083を送る公開APIを持たない。したがってF-3の「Legacy拡張APIが97点以上を送信できる」問題は現行Node-RED版では発生しない。

### F-4【中】iQ-F(FX5)のビット単位一括読出/書込 上限3584点

**マニュアル根拠**: SH-082624 §38.1 — FX5の0401/1401ビット単位(0001/0081/0003…)は **BIN: 3584点**。SH-080003でも7168点はE71経由のiQ-R/Q/L系ユニットの値で、QnA系は3584点、Aシリーズは256点。

**修正前の実装状況**: 全実装が一律 `DIRECT_BIT_POINT_LIMIT = 7168`（dotnet `SlmpClient.cs:30`、python `core.py:1060`、rust `client_rules.rs:8`、cpp `slmp_minimal.cpp:79`、node-red `client.js:38`）。iq-fプロファイルでも7168点まで許容し、3585〜7168点の要求はFX5に拒否される。

**実機確認（2026-07-03）**: FX5U / 192.168.250.100:1025 TCPで、0401/0001ビット単位読出しをM0から実施。1点は終了コード0000、3584点は終了コード0000・応答データ1792バイト、3600点は終了コードC051・エラー情報`00 FF FF 03 00 01 04 01 00`。FX5のビット単位BIN上限3584点と一致し、3585〜7168点をライブラリが許容する場合はPLC側拒否となる。

**推奨対応**: プロファイル（またはアクセス先種別）でビット上限を切替（iq-f=3584）。ワード960点は全ターゲット共通なので現状維持でよい。

**修正（2026-07-03）**: dotnet / python / rust / cpp / node-red の direct bit 一括読出/書込ガードを、iQ-Fプロファイル時のみ3584点に切替。iQ-R/Q/L等は従来通り7168点、ワード単位は960点のまま。各実装に「3585点を通信前に拒否し、エラーメッセージ/戻り値が3584上限を示す」回帰テストを追加済み。

### F-5【中】4E応答のシリアル番号を要求と照合していない

- **照合あり**: cpp（`slmp_minimal.cpp:1174-1179`で不一致をProtocolError）、node-red（`transport.js`でシリアル別pending管理＝並行要求も正しく対応付け）
- **照合なし**: dotnet（`ParseResponse`）、python（`client.request()`はdecode結果のserialを見ない）、rust（`parse_response`）

タイムアウト→再送後に前の要求の遅延応答が届くと、**現在の要求への応答として誤って受理**する（UDPで顕著。TCPでもタイムアウト後にストリームを読み続ける設計のため起こりうる）。コマンドによっては応答サイズが同じで検出不能＝誤ったデータを返す。

**推奨対応**: 4E時は応答serial≠要求serialのフレームを読み捨てて再受信（または即エラー）。

**実機確認（2026-07-03）**: R120P / 192.168.250.100:1025 TCPで、4E/iQ-R形式のD100 1ワード読出しをserial `1234` で送信。応答はD4フレーム、serial `1234`、終了コード0000、データ`00 00`。正常系では要求serialが応答にコピーされることを確認。

**模擬サーバ確認（2026-07-03）**: dotnet / python / rust で、1要求に対して「不一致serialの正常応答（データ0x1111）」→「一致serialの正常応答（データ0x2222）」を連続送信するテストを追加。修正後はいずれも不一致serialを破棄し、0x2222のみを採用する。これにより、遅延応答の誤帰属が回帰テストで検出可能になった。

**修正（2026-07-03）**: dotnet / python / rust は4E時に要求serialを保持し、D4応答のserialが一致しない場合はフレームを破棄して次応答を待つよう修正済み。pythonは同期/非同期とも対応し、非同期版は並行タスク間で期待serialが混線しないようContextVarで保持。cpp / node-redは既存実装で照合済みのため変更なし。

### F-6【低・修正済み】Python: UDPソケットが未connect

修正前の `slmp/client.py` はUDP時に`connect()`せず`sendto/recvfrom`を使い、受信データグラムの送信元を検証しないため、任意ホストからのデータグラムを応答として受理しうる状態だった。dotnet（`UdpClient.Connect`）・rust（`UdpSocket::connect`）はOS側でフィルタされる。

**修正（2026-07-03）**: python同期クライアントはUDPでも `socket.connect((host, port))` を実行し、送信は `send()`、受信は `recv()` に変更済み。これによりOSの接続済みUDPフィルタで、PLC以外の送信元からのデータグラムを応答キューに入れない。非同期クライアントは修正前から `create_datagram_endpoint(..., remote_addr=(host, port))` の接続済みUDP endpoint を使っているため追加修正なし。

**回帰テスト**: 正規UDPサーバが要求を受けた直後、別ポートの偽ソケットからデータ`0x1111`の正常応答風フレームを先に送信し、その後で正規サーバからデータ`0x2222`を送るテストを追加。修正後は`0x2222`だけを採用する。

### F-7【低・記録済み】ZRデバイス番号の公式マニュアル誤記

SH-080003 §8.1のデバイスコード一覧ではZR（連番アクセス方式）の表記が**16進**と読めるが、これは公式マニュアル側の誤記として扱う。全5実装は**10進**でパースし、実機ライブ検証、GX Works3表示、別PLC環境の`ZR1535996`キャプチャ/再検証はいずれも10進動作と一致している。`internal_docs/maintainer/manual_implementation_differences.md` 項目8にも「公式マニュアル表の誤記、実装は10進が正」として記録済み。

### F-8【低・修正済み】異常応答のエラー情報部を構造化

修正前は、終了コード≠0の応答データ部に含まれるエラー情報（ネットワークNo.・局番・要求先I/O・マルチドロップ・コマンド/サブコマンド）を、全実装とも構造化していなかった。python/node-redは生データを例外に添付、dotnet/rust/cppは終了コード中心の診断だったため、PLCがどの要求を拒否したかを追うには手動デコードが必要だった。

**修正（2026-07-03）**: 5実装すべてで、9バイト以上の異常応答データを構造化するよう修正済み。
- python: `SlmpErrorInfo` を追加し、`SlmpResponse.error_info` / `SlmpError.error_info` で参照可能。
- dotnet: `SlmpErrorInfo` を追加し、`SlmpError.ErrorInfo` で参照可能。
- rust: `SlmpErrorInfo` を追加し、`SlmpError.error_info` で参照可能。
- cpp: `SlmpErrorInfo` と `hasLastErrorInfo()` / `lastErrorInfo()` を追加。
- node-red: `parseSlmpErrorInfo()` を追加し、`response.errorInfo` / `SlmpError.errorInfo` で参照可能。

**回帰テスト**: FX5U上限超過時に得た `00 FF FF 03 00 01 04 01 00` を使い、`network=0x00, station=0xFF, module_io=0x03FF, command=0x0401, subcommand=0x0001` として読めることを各実装で確認。
