# Profile Tables Style

This document defines the shared user-manual style for generated `tables/*.md`
profile references across the PLC communication profile repositories.

The SLMP generated tables are the canonical shape. Host Link and Computer Link
tables should match the structure and reading conventions while keeping their
own protocol terms and columns.

## Header Standard

Each generated table page starts with:

1. `# <page title>`
2. `<!-- Generated by tools/generate_profile_tables.py. Do not edit manually. -->`
3. A `Generated user-facing reference. Source: ...` line naming the canonical JSON source files.
4. A schema/date line for every canonical JSON source used by that page.

The exact source line can use the repository's path style, but it must be clear
which JSON file generated the page.

## Purpose Section

Every generated table page includes `## Purpose` near the top.

The purpose text should say what the page helps users compare and what it does
not guarantee. Profile tables are user-facing manual pages, not live PLC
communication logs. Range tables are catalog data for profile selection,
diagnostics, setup, and application-layer validation; they are not runtime
send/receive guards or guarantees that a configured PLC will accept every
address.

## Section Skeleton

Use this order where the data exists:

1. `## Profile Summary`
2. Definition tables such as `## Device Definitions`, `## Area Definitions`, or
   domain-specific option definitions
3. Comparison matrices
4. Policy or note sections when useful
5. `## Appendix: How To Read Cells`

`Profile Summary` includes at least profile ID, display name, and
`Verified models`. If a profile has no existing live-device evidence, use
`Unconfirmed`.

## Comparison Table Direction

Comparison tables use rows for devices, areas, features, limits, or policies
and columns for profiles.

Do not emit long row-oriented tables that repeat the same profile ID for every
device or area when the information can be compared as a matrix.

## Cell Notation

Cells use plain values without Markdown code spans. Empty strings are not
allowed in generated table cells; use `-` for missing, unavailable, or not
applicable values.

State/source cells use the compact form `state/source` when a capability
decision and evidence source are both present. Keep source vocabulary such as
`live`, `policy`, `spec`, `inferred`, and `not-adopted` stable and define it in
the appendix.

## Domain-Specific Columns

Do not copy SLMP column names into other protocols where they do not apply.

Host Link keeps KV-specific terms such as notation, source label, and native or
XYM variants. Computer Link keeps TOYOPUC-specific terms such as direct ranges,
prefixed ranges, packed-word support, width, and step.

If an implementation or source JSON field has an internal snake_case name,
translate it to a user-readable label in the main table. The raw field name may
be preserved in an appendix mapping when it helps maintainers connect the table
back to JSON.

## Appendix: How To Read Cells

Every generated table page includes an appendix explaining:

- The meaning of `-`
- Any status values such as `yes`, `no`, `supported`, `blocked`, or
  `unsupported`
- Any source values such as `live`, `policy`, `manual`, `inferred`, or
  `not-adopted`
- Protocol-specific cell formats such as range syntax, packed overrides, SD
  register references, or native/XYM notation
